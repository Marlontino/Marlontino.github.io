Weather Automation Pipeline

End-to-end weather ETL with PostgreSQL upserts, SQLAlchemy, and a Dockerized one-command stack

Project Overview

An end-to-end data automation workflow that fetches real-world weather data from the Open-Meteo API, persists it in PostgreSQL with idempotent upserts, generates daily summaries directly from the database, and produces visualizations. The entire stack — Postgres plus the pipeline — runs with a single docker-compose up.

Key Features

  • Automated Data Collection: Fetches hourly weather data from the Open-Meteo API and archives the raw JSON locally for replay
  • PostgreSQL-backed Storage: Two tables (hourly_weather, daily_summary) with composite primary keys on (timestamp, lat, lon) and (date, lat, lon)
  • Idempotent Upserts: Every write uses INSERT … ON CONFLICT DO UPDATE, so re-running the pipeline refreshes rows instead of duplicating or erroring
  • DB-native Summaries: Daily averages, mins, maxes, and totals are computed from the database — not from CSV intermediates
  • Visualization: Line plot for daily avg/min/max temperature (°F) and a bar plot for daily precipitation (mm), saved as timestamped PNGs
  • Containerized Stack: docker-compose up brings up Postgres and the pipeline; the pipeline waits on a Postgres healthcheck before running
  • Persistent Volumes: Named weather_pgdata volume preserves the DB across restarts; data/ is bind-mounted so JSON and plots stay visible on the host

How It Works

  1. Fetch: fetch_weather.py hits the Open-Meteo API and archives the raw JSON
  2. Process & Upsert: process_weather.py parses the JSON and upserts hourly rows into Postgres
  3. Summarize: daily_summary.py aggregates hourly rows directly in SQL and writes daily_summary
  4. Visualize: visualize_summary.py renders temperature and precipitation plots from the daily_summary table
  5. Orchestrate: main.py runs the full pipeline; db.py centralizes the SQLAlchemy engine, schema, and upsert helpers

Quick Start (Docker)

$ git clone https://github.com/Marlontino/Weather.git
$ cd Weather
$ docker-compose up --build

That command builds the pipeline image, starts a postgres:16-alpine container with the weather database, waits for Postgres to be healthy, and runs the pipeline once. Inspect results with:

$ docker-compose exec db psql -U weather -d weather \
    -c "SELECT * FROM daily_summary ORDER BY date;"

Local (non-Docker) Setup

# Requires a local Postgres 13+ with a 'weather' DB and user
$ python -m venv venv

$ source venv/bin/activate  # macOS / Linux
$ pip install -r requirements.txt

$ python -m src.main

Database Schema

CREATE TABLE hourly_weather (
    timestamp        TIMESTAMP NOT NULL,
    latitude         NUMERIC(8,5) NOT NULL,
    longitude        NUMERIC(8,5) NOT NULL,
    temperature_c    DOUBLE PRECISION,
    precipitation_mm DOUBLE PRECISION,
    PRIMARY KEY (timestamp, latitude, longitude)
);

CREATE TABLE daily_summary (
    date                 DATE NOT NULL,
    latitude              NUMERIC(8,5) NOT NULL,
    longitude           NUMERIC(8,5) NOT NULL,
    avg_temperature_c      DOUBLE PRECISION,
    min_temperature_c      DOUBLE PRECISION,
    max_temperature_c      DOUBLE PRECISION,
    total_precipitation_mm DOUBLE PRECISION,
    PRIMARY KEY (date, latitude, longitude)
);

Schemas are created automatically on first run by db.init_schema().

How Upserts Work

The pipeline uses PostgreSQL's INSERT … ON CONFLICT DO UPDATE to stay idempotent across re-runs:

INSERT INTO hourly_weather (timestamp, latitude, longitude, temperature_c, precipitation_mm)
VALUES (...)
ON CONFLICT (timestamp, latitude, longitude)
DO UPDATE SET
    temperature_c    = EXCLUDED.temperature_c,
    precipitation_mm = EXCLUDED.precipitation_mm;

This means the pipeline can run on a cron/scheduled container multiple times a day with overlapping windows — no duplicates, no primary-key violations, just refreshed rows.
